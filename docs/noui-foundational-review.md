# NoUI enterprise platform best practices: a comprehensive technical blueprint

**NoUI sits at the intersection of three demanding contexts — AI-augmented development, pension administration mathematics, and government compliance — requiring an unusually disciplined foundation before a single line of production code is written.** Colorado PERA's pension administration system is **40 years old, the oldest still in use by any U.S. public pension**, processing $5.4B in annual benefit payments across 700,000+ members and 500+ employer agencies. This modernization opportunity is massive but unforgiving: errors in pension calculations have fiduciary consequences, Colorado has the nation's strictest state accessibility law, and SOC 2 Type II plus NIST 800-53 compliance must be architected from day one rather than bolted on later. The research below synthesizes 2024–2025 best practices across nine domains into a single implementation-ready reference, with every recommendation grounded in the specific constraints of pension middleware targeting public pension administration systems.

## 1. Claude Code and AI-augmented development demand rigorous specification discipline

The single highest-leverage investment for a Claude Code-driven build is the **CLAUDE.md file** — Anthropic's persistent context mechanism loaded at the start of every session. Research from HumanLayer, Anthropic's internal teams, and enterprise practitioners converges on a clear set of principles: keep the file **under 200 lines**, include only universally applicable instructions, and use progressive disclosure by referencing deeper documentation stored in an `agent_docs/` directory. Frontier reasoning models can reliably follow roughly 150–200 instructions, and Claude Code's system prompt already consumes about a third of that capacity. Every irrelevant instruction degrades attention to important ones.

For NoUI specifically, the CLAUDE.md should contain exact build/test/lint commands, a concise architecture map, **pension domain terminology** (member, scheme, annuity, vesting, actuarial), strict TypeScript rules (no `any` types, named exports only), and pointers to deeper context documents for API patterns, pension business logic, and testing strategy. Anthropic's internal teams use hierarchical CLAUDE.md files for monorepos — root-level plus package-level — which aligns well with NoUI's modular architecture.

**Project structure for autonomous development** follows several patterns validated by Anthropic's engineering teams. Start every task from a clean git state, commit checkpoints at least hourly, and use Plan Mode before any change touching three or more files. Classify tasks by supervision level: peripheral features and prototyping run in auto-accept mode (~70% autonomous), while core pension business logic requires synchronous supervision with detailed prompts. The Writer/Reviewer pattern — one Claude session writes code, a fresh session reviews it — prevents self-reinforcing bias. Git worktrees enable parallel Claude instances on different branches with isolated context.

**Context window management** is the primary constraint on quality. Performance degrades as context fills, so the discipline is aggressive: one feature per chat session, manual `/compact` at 50% context usage, and `/clear` when switching tasks. After two failed corrections, clear and rewrite a better initial prompt rather than polluting context with failed approaches. Pipe data directly (`cat error.log | claude`) instead of asking Claude to search for it.

The emerging practice of **Test-Driven Generation** represents the optimal workflow: humans write failing tests that encode pension business rules and edge cases, Claude implements the minimal code to pass them, humans review for correctness and architecture, then Claude refactors while tests stay green. Tests become the specification document. This inverts the traditional relationship — when AI generates code, tests are MORE important, not less.

**Preventing "vibe coding" drift** requires explicit structural discipline. Academic research identifies five core risks: code generation bloat, comprehension debt (developer doesn't understand AI output), architectural inconsistency, shallow testing, and dependency sprawl. The mitigation framework involves risk-zoning the codebase (categorizing areas by risk level with appropriate AI autonomy permissions), tracking technical debt ratio (warning threshold at >20%), reinvesting time savings into review rather than more generation, and running planned hardening phases between MVP milestones.

For **code review**, the evidence supports a three-layer architecture. Layer 1 handles real-time IDE feedback through linters and type checking. Layer 2 deploys AI-powered PR analysis through Claude Code's GitHub App or CodeRabbit. Layer 3 reserves human architectural review for security-critical paths, pension calculation business logic, data model changes, and any architectural decisions. Qodo's 2025 data shows **25% of developers estimate 1 in 5 AI suggestions contain functional errors** — teams with strong review processes see quality improvements while those without see declines.

## 2. A modular monolith is the right architectural starting point

The 2024–2025 industry consensus has shifted decisively: **microservices benefits only appear with teams exceeding 10 developers**, and below this threshold monoliths consistently outperform. Martin Fowler's "Monolith First" principle, Amazon Prime Video's high-profile migration back to monolithic architecture, and multiple architecture publications all converge on the same recommendation for small, scaling teams.

NoUI should adopt a **modular monolith with evolutionary architecture**, where module boundaries map to pension domain bounded contexts: Member Services, Benefit Calculation, Contribution Processing, Employer Reporting, Payment Processing, Document Management, and AI Services. Each module owns its data access through schema separation within a shared PostgreSQL database, communicates in-process via well-defined interfaces (not HTTP), and is designed so any module can be extracted as a microservice later via the Strangler Fig pattern. This decision must be made before development begins — it shapes everything downstream.

**API design should be contract-first REST using OpenAPI 3.x**, not GraphQL. REST is the standard for financial and government interoperability, Financial-grade API (FAPI) standards are built on REST/OAuth, and government systems expect REST interfaces. Use Microsoft's **TypeSpec** as a higher-level language to generate OpenAPI specs, and AsyncAPI for event-driven interfaces. API security requires OAuth 2.0 with PKCE, OpenID Connect, mutual TLS for service-to-service communication, and complete audit logging of all API access.

The **data layer** is the most critical architectural decision. NoUI must integrate with PERA's 40-year-old mainframe PAS, the STARS employer portal, and other pension plan systems. The Anti-Corruption Layer pattern — façades for data access, adapters for protocol translation, translators for model mapping — ensures NoUI's domain models are never polluted by legacy data structures. Data synchronization will likely begin as batch synchronization (nightly/hourly) for mainframe data, evolving toward Change Data Capture for near-real-time sync as capabilities allow. The Strangler Fig pattern enables gradual migration, starting with read-only integration and progressively moving write operations.

**Event-driven architecture** is strongly recommended for pension financial systems. Financial institutions implementing EDA report **47% improvement in fraud detection** and 31% reduction in false positives. For NoUI, apply CQRS (separate read/write models) for pension data where reads vastly outnumber writes, selective Event Sourcing for audit-critical domains like benefit calculations and payment processing, and the Saga pattern (orchestration-based, using Temporal.io) for multi-step processes like retirement processing and benefit recalculation. Start with RabbitMQ for simplicity, migrate to Kafka when event streaming volume demands it.

The recommended technology stack centers on **Node.js/TypeScript for full-stack consistency**, React with Next.js App Router for the frontend, shadcn/ui with Tailwind CSS for accessible components, PostgreSQL for the database, Keycloak for on-premises authentication, and the OpenTelemetry plus Grafana LGTM stack for observability. Infrastructure as Code via Terraform and CI/CD through GitHub Actions should be established from day one.

## 3. Security and compliance architecture cannot be retrofitted

NoUI should be classified as a **NIST FIPS 199 MODERATE impact system** at minimum, requiring **287 controls** across 20 NIST 800-53 control families. Given that pension data includes SSNs, financial account information, and benefit calculations affecting fiduciary obligations, the security architecture must be established before development begins.

**SOC 2 Type II readiness** requires specific controls from day one that are extremely difficult to retrofit: centralized identity provider with MFA enforced for all users, version control with branch protection and signed commits, AES-256 encryption for data at rest and TLS 1.3 for data in transit, HashiCorp Vault for secrets management, written security policies (Information Security Policy, Acceptable Use Policy, Incident Response Plan, Data Classification Policy), a formal risk register, and centralized logging with security event alerting. SOC 2 Type I audit should be targeted within 3–6 months of initial build, with Type II following after 3–6 months of operating Type I controls. All five Trust Services Criteria apply to pension middleware: Security, Availability, Processing Integrity, Confidentiality, and Privacy. Compliance automation platforms like **Vanta, Drata, or Secureframe** can cut audit preparation time by up to 90%.

The **most critical NIST 800-53 control families** for pension middleware are Access Control (AC — SSN/PII access must be role-restricted), Identification and Authentication (IA — MFA mandatory per IA-2), Audit and Accountability (AU — immutable audit trails for fiduciary data), System and Communications Protection (SC — the largest family, covering encryption and transmission security), and System and Information Integrity (SI — input validation, flaw remediation, system monitoring). Tier 2 priorities include Risk Assessment, Configuration Management, Incident Response, and Contingency Planning.

**Authentication and authorization** should use a hybrid RBAC plus ABAC model. RBAC provides coarse-grained roles (pension_admin, employer_hr, member_self_service, auditor, system_admin), while ABAC handles fine-grained context-aware decisions like employer-based data isolation, time-based access restrictions, and data classification-based access controls for SSN fields. Zero Trust Architecture per NIST SP 800-207 should be adopted across all five CISA pillars: Identity, Devices, Networks, Applications, and Data.

**Security in the AI-augmented development pipeline** introduces specific risks: approximately 40% of AI-generated suggestions may contain potential vulnerabilities. Controls include Claude Code permission configuration that denies access to secrets and environment files, mandatory human code review for all security-sensitive paths, AI-specific SAST scanning via Semgrep in CI/CD, sandboxed execution environments, a deterministic dependency allowlist, and prohibition against pasting real pension data into Claude Code contexts. The application security tooling stack should include **Semgrep for SAST, Snyk for SCA, OWASP ZAP for DAST, GitLeaks for secrets detection, and Trivy for container scanning**, all integrated into the CI/CD pipeline.

**Colorado-specific requirements** are stringent. The Colorado Data Breach Notification Law requires **30-day notification** to affected residents after breach discovery and notification to the Attorney General for breaches affecting 500+ residents. The Colorado Privacy Act requires Data Protection Assessments before processing that presents heightened risk, consent for processing sensitive data including SSNs and financial data, and carries penalties of **$2,000–$20,000 per violation** with no cure period as of January 2025.

## 4. Testing strategy anchors on financial calculation accuracy and accessibility

The testing pyramid for AI-augmented builds shifts emphasis toward unit tests as the base: roughly **55% unit tests** (financial calculations and business logic), 25% component tests (Storybook plus Vitest), 15% integration tests (API contracts and data flows), and 5% E2E tests (critical pension workflows). Every AI-generated function requires a human-written test that predates the code.

**Pension math testing** is the highest-priority testing area. Financial calculations must be provably correct through multiple reinforcing approaches. Golden file testing maintains a spreadsheet of known-correct calculations from the actuarial team — every function must produce results matching reference values to the penny. Decimal precision testing using libraries like `decimal.js` or `big.js` (never floating-point) verifies calculations maintain precision through multi-step operations. Boundary testing covers maximum/minimum service years, salary caps, age thresholds, fiscal year transitions, and leap years. Cross-validation implements the same calculation two different ways and verifies identical results.

**Property-based testing with fast-check** generates hundreds of random inputs to verify mathematical invariants that must always hold: benefits are always non-negative, benefits increase monotonically with years of service, contributions never exceed salary. **Mutation testing with Stryker** is essential for financial modules — it introduces small code changes (replacing `>` with `<`, `+` with `-`) and verifies tests catch them. A single operator error could miscalculate millions in benefits. Target mutation score of **≥90% for calculation modules**, run incrementally in CI for changed files and fully nightly.

**Accessibility testing** is non-negotiable. Colorado HB 21-1110, effective July 2024, makes Colorado the **first state to mandate that all state and local government websites meet accessibility standards**, with penalties of **$3,500 per violation**. The federal ADA Title II rule finalized in April 2024 requires WCAG 2.1 Level AA compliance by **April 24, 2026**. Target WCAG 2.2 AA for future-proofing. Automated testing through axe-core catches approximately 57% of issues; integrate `@axe-core/playwright` into E2E tests and `@storybook/addon-a11y` into component development, with zero tolerance for new violations in CI. Manual testing with screen readers, keyboard navigation, and real users with assistive technologies covers the remaining 43%.

**Coverage thresholds** should be tiered: 95% line and branch coverage for financial calculations with 100% function coverage, 90% for general business logic, 80% for React components, and 80% overall minimum. Quality gates in CI/CD enforce eight sequential checks: lint and type check, unit and component tests with coverage, accessibility scan, security scan, bundle size budget, E2E tests, performance (Lighthouse CI), and mutation score for financial modules.

The recommended tooling stack is **Vitest** for unit and component testing, **Playwright** for E2E and accessibility, **Storybook 8** with a11y and MSW addons for component development, **MSW 2.x** for network-level API mocking reusable across all test types, **fast-check** for property-based testing, and **Stryker** for mutation testing. All configured before development begins.

## 5. Performance planning starts with specific numeric targets

Performance budgets for NoUI should target **99.95% availability** (≤4.38 hours downtime/year), **API response time under 500ms at p95** for standard CRUD operations, **pension calculation response under 2 seconds at p95** for complex actuarial calculations, and **batch processing completion within a 4-hour window** for nightly contribution reporting and data sync. Recovery Time Objective should be under 1 hour with Recovery Point Objective under 15 minutes.

Core Web Vitals targets for the admin platform should exceed Google's "good" thresholds: **LCP under 1.8 seconds** (admin dashboards with data tables), **INP under 150ms** (critical for form-heavy pension data entry), and **CLS under 0.05** (financial data tables must be visually stable with zero shift during loading). Maximum JavaScript bundle should be 300 KB gzipped for initial load.

**Caching strategy** follows a tiered architecture. L1 is in-memory application cache (sub-millisecond). L2 is Redis (sub-5ms, recommended for enterprise with distributed caching support for hybrid cloud/on-prem). Actuarial tables cache for 24 hours to 30 days, contribution rate tables cache until legislative change with event-driven invalidation, member benefit estimates cache for 1 hour keyed by member_id plus calculation parameters, and static reference data pre-fetches in-memory for 24 hours. Critical rules: **never cache intermediate calculation results** that could become stale, use version-based cache keys, implement event-driven invalidation when member data changes, encrypt all cached PII with AES-256, and log all cache hits and misses for financial calculations per GASB 67/68 audit requirements. Proper caching can reduce response times by 90% and server load by 70%.

**Load testing** with Grafana k6 should model Colorado pension system scale: typical concurrent users of 500–2,000, peak concurrent users of 5,000–10,000 during year-end reporting (December–January, 5–8× normal load), open enrollment (3–5× normal), and monthly payroll reporting deadlines (2–3× normal). Six test types are needed: smoke, load, stress, spike, soak (4 hours for memory leak detection), and endurance (24 hours for overnight batch plus daytime concurrent use).

The monitoring and observability stack should be **OpenTelemetry plus the Grafana LGTM stack** (Loki for logs, Grafana for visualization, Tempo for traces, Mimir for metrics). This avoids vendor lock-in (critical for government procurement), supports self-hosted deployment for data sovereignty, and uses OpenTelemetry's vendor-neutral instrumentation standard. Key metrics to track include pension calculation latency at p50/p95/p99, cache hit ratio (target >95% for reference data, >80% for calculations), database connection pool utilization, and replication lag for read replicas. Implement tail sampling to capture 100% of errors and slow traces while sampling routine traffic at 10–20%, and **redact SSNs and salary data in the telemetry pipeline** using OpenTelemetry Collector processors.

Database optimization for legacy integration requires an Anti-Corruption Layer with materialized views on legacy databases for common query patterns, read-optimized replicas for complex pension calculations, table partitioning by year for contribution history, BRIN indexes for time-series data, and **separate connection pools for real-time API versus batch/reporting jobs** to prevent batch operations from starving interactive users during year-end reporting.

## 6. Enterprise UX must earn trust through transparency and rigorous accessibility

Pension administration UX must prioritize **trust through transparency** — every financial action needs clear confirmation steps, audit trails, and status indicators. Progressive disclosure presents key financial data upfront while making detailed reports accessible through clearly labeled secondary actions. Role-based interfaces serve four distinct user types: pension administrators (processing queues, exception reports, system oversight), employer reporters (contribution submission, filing deadlines, member management), retirees (payment history, tax documents), and members (benefit estimates, contribution history).

The design system should follow a **three-layer token architecture**: primitive tokens (raw color, spacing, typography values), semantic tokens (purpose-based references like `color.interactive.default`), and component tokens (specific to individual components). The recommended toolchain is Figma with Tokens Studio for design, Style Dictionary for token translation to CSS variables and JavaScript, and Storybook 8+ for component documentation and testing. Reference the **U.S. Web Design System (USWDS)** for government-adjacent patterns — it serves 94 agencies and 160+ websites with 40+ accessible, mobile-friendly components — and IBM Carbon for enterprise form patterns.

**Desktop-first with responsive adaptation** is the correct approach for pension administration. Administrators and employer reporters work at office desktops during business hours on tasks involving complex data entry, multi-column data tables, and financial calculations — inherently desktop workflows. Design for 1280px+ primarily, with graceful degradation at tablet (768px–1279px) and mobile (<768px) breakpoints. The member portal is the exception — if built, consider mobile-first for that module since members may check pension information on phones.

**Form design** for employer contribution reporting should use a sectioned form (not wizard), since pension administrators are repeat users needing full context and non-linear access. Single-column layout eliminates Z-pattern scanning confusion. Autosave prevents data loss during complex reporting. Real-time calculated totals display as data is entered. Validation follows the "reward early, punish late" principle: errors trigger on blur, empty required fields flag only on submit, and errors clear immediately on correction. Financial-specific patterns include auto-formatted currency fields, masked SSN inputs, constrained percentage fields, and cross-field validation ensuring contribution totals match component sums.

**Data visualization** for pension dashboards should use line charts for contribution trends, grouped bar charts for budget-versus-actual comparisons, KPI tiles with sparklines for dashboard summaries (total members, contributions, funded ratio), and waterfall charts for how contributions build to total fund value. Use Recharts or Nivo for React-based charting with accessibility built in, and TanStack Table or AG Grid for data tables.

Information architecture should be validated with **tree testing before implementation** — Smart Pension validated IA changes with 200 participants at approximately $10 per person and got results in 4–5 hours. Persistent sidebar navigation for desktop, breadcrumbs for deep pages, and global search with typeahead supporting member lookup by name, SSN, or employer.

## 7. AI modules must never touch deterministic pension calculations

The fundamental principle for AI in pension administration is a clear boundary: **pension benefit calculations, contribution rate computations, vesting determinations, actuarial computations, tax withholding, and payment processing must be purely deterministic**. LLMs are nondeterministic by nature — even with temperature set to zero, outputs may vary. As one financial analysis expert states: "Given the state of the risk factors, finance becomes a simple, deterministic mathematical exercise. Nothing to be gained with AI."

AI adds genuine value in six specific areas. **Member service chatbots** handle routine inquiries about balances, eligibility, and contribution history — Bank of America's Erica handled 1B+ interactions from 32M users, and pension providers are deploying similar capabilities. **Data cleansing and reconciliation** across PERA's fragmented systems can reduce manual cleanup by 50–70%, critical for legacy migration. **Document processing** using NLP extracts key data from employer contribution reports, compressing month-end processes from days to hours. **Anomaly detection** flags unusual employer contribution patterns, duplicate payments, and salary-to-contribution mismatches using Isolation Forest and Autoencoder algorithms with per-employer historical baselines. **Predictive analytics** anticipates early retirement likelihood, refund requests, and processing surges. **AI-assisted form completion** pre-populates known data and provides intelligent field suggestions.

The recommended integration architecture is **RAG plus Function Calling**: the LLM interprets natural language intent, retrieves relevant pension rules via RAG, invokes deterministic APIs for actual calculations, and formats the response with citations. Every numerical answer must come from deterministic database queries, never from the LLM. Temperature should be zero for data queries, and confidence scoring should route low-confidence responses to human agents.

**Responsible AI governance** must follow the U.S. Treasury's Financial Services AI Risk Management Framework released in February 2025, which specifies **230 control objectives** across governance, data, model development, validation, monitoring, and third-party dependencies. NIST's AI Risk Management Framework provides the structural foundation with four functions: Govern, Map, Measure, Manage. Every AI-driven recommendation must be traceable to source data using explainability techniques like SHAP values, all AI decisions must be logged in WORM-style storage for auditability, and clear disclosure is required when members interact with AI versus human agents.

A **risk-based human-in-the-loop framework** governs AI autonomy: Tier 1 (no review needed) covers FAQ responses and form suggestions; Tier 2 (async human review) covers anomaly flags and data correction suggestions; Tier 3 (synchronous human decision) covers benefit determinations and eligibility edge cases; Tier 4 (AI excluded) covers calculations that determine payment amounts, appeals, and fiduciary decisions. If AI confidence falls below 85% or multiple models disagree by more than 8%, automatic escalation to human review occurs.

## 8. Program management shifts from tracking effort to managing review throughput

When AI generates most code, **the bottleneck shifts from development to review, testing, and integration**. The PM role transforms from tracking developer hours to orchestrating human-AI collaboration. Organizations that increased AI development output experienced new bottlenecks precisely in these downstream activities.

**One-week sprints** are recommended over two-week cycles. AI compresses the generation phase so dramatically that two-week sprints create inventory waste. The proposed structure front-loads specification: Monday for planning and specification writing (create `plan.md` files for each story), Tuesday–Wednesday for AI generation with parallel human review, Thursday for integration testing and compliance verification, Friday for deployment to staging, sprint review, and retrospective. Front-load **20–30% of sprint time in planning** versus the traditional 5–10%, because specification quality directly determines AI output quality. Keep WIP limits strict at 2–3 features in review simultaneously.

**Velocity tracking** must separate generation velocity from delivery velocity. Claude may generate a feature in 30 minutes, but review, testing, and integration may take 2 days. Track feature cycle time (end-to-end from spec to merged and tested), review throughput (PRs reviewed per sprint), AI generation-to-approval ratio (measuring spec quality), and Definition of Done compliance rate. Use Cumulative Flow Diagrams to visualize where work accumulates — expect the bottleneck in "In Review" rather than "In Development." Report to stakeholders on outcomes (pension workflows enabled), not output (lines of code generated).

The **Definition of Done** for AI-generated code must be MORE rigorous, not less. It spans five categories: planning (spec completed and annotated by domain expert, pension rules verified against regulations), code quality (passes static analysis, follows CLAUDE.md conventions, reviewed by human with domain context), testing (minimum 80% coverage for new code, pension calculations verified with known test cases, edge cases documented), security and compliance (no hardcoded secrets, input validation implemented, PII handling verified, regulatory compliance confirmed), and documentation (API docs updated, ADRs updated, AI attribution noted in PR).

**PMO tooling** recommendation for NoUI at current scale: **Linear for sprint management** (fastest UI, developer-first, excellent GitHub integration), **GitHub for code and PR management** (Claude Code integrates natively), and **Notion for knowledge base and specifications**. Jira's configuration overhead is counterproductive for small teams, though if government stakeholders specifically require it, compliance trumps preference.

**Team structure** for Phase 1 (3–5 people) centers on a Technical Lead/Architect who is the primary Claude Code operator and reviews all AI output, a PM/Scrum Master managing the review pipeline, a part-time Pension Domain Expert validating business rules, and a QA/Test Engineer reviewing AI-generated tests and maintaining the compliance test suite. The highest-leverage role to establish early is an **"AI Workflow Architect"** who owns CLAUDE.md configuration, prompt templates, workflow patterns, and continuously optimizes human-AI collaboration. Scale to 5–10 people by adding a second developer, DevOps engineer, and full-time domain expert as Product Owner.

## 9. Code quality and technical debt prevention require structural guardrards

**Biome** (Rust-based, 10–25× faster than ESLint plus Prettier combined) is the recommended primary linting and formatting tool for TypeScript/React, configured as a Claude Code Hook rather than a CLAUDE.md instruction. This creates an automated enforcement loop: Claude generates code, the hook runs Biome on stop, and errors are presented for Claude to fix immediately. Configure `noExplicitAny: "error"` and `noUnusedVariables: "error"` as minimum rules.

**Architectural Decision Records** are especially critical in AI-assisted development because AI doesn't understand WHY decisions were made. Without ADRs stored in `docs/adr/`, Claude may undo intentional architectural choices on subsequent sessions. Add to CLAUDE.md: "Before modifying the data layer, read docs/adr/ for existing architectural decisions." Claude can draft ADRs from provided context, but humans must validate and approve them. Use fitness functions — automated checks verifying ADR decisions are maintained — as enforcement.

**Dependency management** requires a layered security approach. Disable npm lifecycle scripts globally (`npm config set ignore-scripts true`) to prevent supply-chain attacks, commit lock files for deterministic installs, scan with Snyk for vulnerabilities, use Socket.dev for supply-chain attack detection beyond CVEs, and run `depcheck` regularly to remove unused dependencies. Generate a Software Bill of Materials for every release. Maintain an approved dependency allowlist that Claude Code must respect — block the AI from importing arbitrary packages.

**Technical debt tracking** requires explicit structural allocation. Track Technical Debt Ratio via SonarQube with a warning threshold at 20%. Dedicate **20% of sprint capacity to refactoring** and do not cut this when AI speeds up generation — the discipline must be that time savings from faster generation are reinvested into review and refactoring, not more features. Monitor the refactoring-to-feature ratio: if code generation rises while refactoring falls, intervene immediately. Run regular AI debt audit prompts asking Claude to analyze the codebase for duplication, inconsistent patterns, missing tests, dead code, and outdated dependencies.

**Documentation standards** for AI-generated codebases follow a hierarchy: CLAUDE.md at the top level (project conventions), progressive disclosure docs in `agent_docs/` (task-specific guidance), ADRs in `docs/adr/` (architectural decisions), OpenAPI specs generated from types (API documentation), README per package or service (purpose, setup, key patterns), and inline comments for WHY, not WHAT. End every Claude Code session by asking Claude to summarize work and suggest documentation updates — this is Anthropic's internal practice for continuous documentation improvement.

## Implementation sequencing: what to establish before writing production code

The following items represent **non-negotiable foundations** that must be in place before active development begins, ordered by dependency and criticality.

**Week 0–1 (Security and governance foundation):** Written security policies, centralized IdP with MFA, HashiCorp Vault for secrets management, Git repository with branch protection and signed commits, GitLeaks pre-commit hooks, Claude Code permission configuration, data classification schema for pension data types, risk register, AI Usage Policy document for government stakeholder confidence, and Colorado compliance documentation plan.

**Week 1–2 (Development infrastructure):** CLAUDE.md file carefully handcrafted under 200 lines with pension domain terminology, Biome configuration for TypeScript/React, Claude Code Hooks for pre-commit test gates and lint/format enforcement, TypeScript strict mode, ADR template and directory, CI/CD pipeline with lint → type-check → test → security scan gates, Vitest configured with coverage thresholds, MSW handler structure, Playwright configured with axe-core, and the decimal precision library selected for financial math.

**Week 2–3 (Architecture and design decisions):** Modular monolith with bounded context boundaries defined, contract-first REST API specs using OpenAPI 3.x, Anti-Corruption Layer design for legacy system integration, OAuth 2.0 plus OIDC authentication architecture, RBAC/ABAC authorization model with pension-specific roles, audit logging strategy (immutable, tamper-proof), event-driven architecture contracts via AsyncAPI, database schema with connection pooling and partitioning strategy, Redis caching architecture with TTL policies and key naming conventions, and performance budgets with SLA definitions.

**Week 3–4 (Design system and UX foundation):** Three-layer design token architecture with naming conventions, WCAG 2.2 AA accessibility target confirmed, desktop-first responsive strategy with breakpoint definitions, Storybook 8 setup with a11y addon and MSW integration, component naming conventions and story organization, information architecture per user role validated with tree testing, form pattern standards including validation timing and error messages, and data visualization standards.

**Week 4+ (Progressive additions as development proceeds):** Progressive disclosure docs added as patterns emerge, custom Claude Code slash commands built as workflows stabilize, SonarQube/SonarCloud for comprehensive quality analysis, Stryker mutation testing on financial calculation modules, k6 performance tests for API endpoints, Lighthouse CI for frontend performance budgets, visual regression testing via Chromatic, SOC 2 Type I readiness assessment at month 3–6, quarterly manual accessibility audits, and AI PR review tooling after initial codebase establishes patterns.

## Pension administration context demands higher standards everywhere

Several cross-cutting considerations unique to the Colorado pension and government-adjacent context elevate requirements across all nine domains. Colorado PERA's **$5.4B in annual benefit payments** and service to tens of thousands of public employees mean that every calculation error has direct financial consequences for real people's retirements. The platform processes PII at massive scale — SSNs, salary histories, bank account numbers for direct deposit — under Colorado's uniquely strict data protection and accessibility laws.

The fiduciary obligation under Colorado Revised Statutes Title 24, Article 51 means that AI governance, calculation accuracy, and audit trails are not optional enhancements but legal requirements. NoUI must demonstrate compliance to PERA's 16-member Board of Trustees, the Pension Review Commission, and potentially the Colorado Attorney General. Every architectural decision, from caching strategy to AI module boundaries, must be defensible in this context.

The modernization opportunity is real — PERA approved a 20% budget increase to $132M and is hiring 70 new staff focused on technology modernization — but the 5–10 year modernization timeline means NoUI must coexist with legacy systems for years, making the Anti-Corruption Layer and Strangler Fig patterns not just best practices but survival requirements. The middleware must prove its value through incremental integration, starting with read-only access and progressively taking on more responsibility as trust is established.

Building on these foundations with disciplined AI-augmented development, rigorous testing of financial calculations, and compliance-first security architecture positions NoUI to serve as the modern middleware layer that Colorado's aging pension infrastructure urgently needs — while meeting the highest standards that public trust demands.
