# Two-Source Proof — Design Document

**Goal:** Run the full migration pipeline end-to-end against two structurally different
legacy pension databases (PRISM and PAS), proving the engine can profile, map, transform,
load, and reconcile real source data into the NoUI canonical schema.

**Gate:** Phase 3 exit. Unlocks Phase 4 (Production Hardening).

**Approach:** Full database isolation — separate PostgreSQL containers for each source system.

---

## 1. Docker Architecture

Two new PostgreSQL containers added to `docker-compose.yml`:

```yaml
prism-source:
  image: postgres:16
  environment:
    POSTGRES_DB: prism_prod
    POSTGRES_USER: prism
    POSTGRES_PASSWORD: prism
  volumes:
    - ./migration-simulation/sources/prism/init:/docker-entrypoint-initdb.d
  networks:
    - default
  # No host port mapping (per Port Management Phase 1)

pas-source:
  image: postgres:16
  environment:
    POSTGRES_DB: pas_prod
    POSTGRES_USER: pas
    POSTGRES_PASSWORD: pas
  volumes:
    - ./migration-simulation/sources/pas/init:/docker-entrypoint-initdb.d
  networks:
    - default
  # No host port mapping
```

Source databases are internal-only. The migration service connects via Docker DNS
(`prism-source:5432`, `pas-source:5432`) using the `SourceConnection` JSONB config
stored on each engagement.

### Port Registry Update

Add to `infrastructure/ports.env`:
```
PRISM_SOURCE_PORT=5432    # Internal only (prism-source container)
PAS_SOURCE_PORT=5432      # Internal only (pas-source container)
```

### PRISM DDL Adaptation (SQL Server → PostgreSQL)

The PRISM schema is written in SQL Server syntax. Changes needed:

| SQL Server | PostgreSQL |
|------------|-----------|
| `GO` delimiter | Remove |
| `DATETIME` | `TIMESTAMP` |
| `VARCHAR(n)` | `VARCHAR(n)` (same) |
| `DECIMAL(p,s)` | `NUMERIC(p,s)` |
| `CHAR(n)` | `CHAR(n)` (same) |
| `SMALLINT` | `SMALLINT` (same) |
| `USE PRISM_PROD;` | Remove (handled by container env) |
| Schema-less tables | Wrap in `CREATE SCHEMA src_prism;` |

PRISM tables (from original DDL):
- `PRISM_MEMBER` — core member record (500 rows)
- `PRISM_MEMBER_ADDR` — address history
- `PRISM_EMP_SPELL` — employment spells
- `PRISM_JOB_HIST` — job/compensation changes
- `PRISM_SAL_ANNUAL` — pre-1998 annual salary (OPUS migration artifact)
- `PRISM_SAL_PERIOD` — post-1998 per-period salary detail
- `PRISM_CONTRIB` — employee contributions
- `PRISM_CONTRIB_LEGACY` — pre-1998 contributions (OPUS)
- `PRISM_SVC_CR_BAL` — service credit running balance
- `PRISM_BENEFIT_CALC` — benefit calculations (stored inputs + results)
- `PRISM_BENEFICIARY` — beneficiary/QDRO records
- `PRISM_EMPR_LIST` — employer reference
- `PRISM_MISC_CODES` — overloaded code table
- `PRISM_NOTES` — free-text notes

### PRISM Generator Adaptation

The existing `prism_data_generator.py` outputs JSON. Adapt to output SQL INSERT
statements directly:

```python
# Output: migration-simulation/sources/prism/init/02_seed.sql
# INSERT INTO src_prism.PRISM_MEMBER (MBR_NBR, LAST_NM, ...) VALUES (...);
```

Data characteristics that exercise the pipeline:
- 500 members with realistic status distribution
- ~15% BIRTH_DT format errors (MMDDYYYY vs YYYYMMDD)
- SSN format split (dashes pre-2001, no dashes post-2001)
- Overloaded STATUS_CD (6 codes across 3 eras)
- Pre-1998 OPUS migration artifacts (annual-only salary)
- Benefit calculations for ~60% of eligible retirees
- Member key referenced as MBR_NBR/EMPL_ID/EMP_NBR/EMP_ID across tables

The PAS generator outputs CSV; adapt `pas_load_from_csv.sql` to output INSERT

## 4. Test Script

**File:** `scripts/run_two_source_proof.sh`

Orchestrates the full proof run:

```
Phase 1: Infrastructure
  - docker compose up -d --build
  - Wait for: migration, migration-intelligence, prism-source, pas-source

Phase 2: PRISM Migration
  - Create engagement (source_system_name: "PRISM")
  - Configure source (host: prism-source, port: 5432, db: prism_prod, user: prism)
  - Profile source tables (all PRISM_* tables in src_prism schema)
  - Generate mappings (dual strategy)
  - Approve mappings
  - Create and execute batch
  - Run reconciliation (all 3 tiers)
  - Check gate

Phase 3: PAS Migration
  - Same flow with PAS source connection (host: pas-source, port: 5432, db: pas_prod)

Phase 4: Cross-Source Verification
  - Both sources coexist in canonical tables
  - Lineage covers all loaded rows
  - Exception counts consistent
  - Schema unchanged (before/after diff)

Phase 5: Cross-Language Verification
  - Go: go test ./reconciler/... -run TestCrossLanguage
  - Python: pytest tests/test_cross_language.py

Phase 6: Results
  - Output gate results for both sources
  - Generate results document
```

## 7. Risk Register

| # | Risk | Likelihood | Impact | Mitigation |
|---|------|-----------|--------|------------|
| R1 | PRISM DDL port misses SQL Server idioms | Medium | Low | Review all 14 tables systematically |
| R2 | Generator output volume overwhelms Docker init | Low | Medium | Limit to 100 members initially, scale if clean |
| R3 | Profiler SQL queries don't work against source schemas | High | Medium | Source tables use different naming; profiler queries need schema qualification |
| R4 | Template matcher doesn't recognize PRISM column names | Medium | High | PRISM uses non-standard names (MBR_NBR, NATL_ID); may need template additions |
| R5 | Reconciliation formulas don't match generator's benefit calcs | Medium | High | Generator uses same formula as reconciler; verify shared fixtures cover edge cases |
| R6 | Docker init ordering — source DBs not ready when migration service starts | Medium | Low | Migration service retries on source connection; add healthcheck to source containers |
